## Money / Config / Bridge — Drift-Klasse

- **Zwillings-RPCs „die das Gleiche tun" driften über MEHRERE Achsen — beim Anfassen ALLE diffen, nicht nur die berührte** (S413): Zwei Money-RPCs, die dasselbe fachliche Resultat liefern sollen (Markt-Kauf `buy_player_sc` = auto-picked günstigste Order ↔ `buy_from_order` = gewählte Order), driften unabhängig in JEDER nicht-zentralisierten Entscheidung auseinander. Live-Befund 413 = **4 Achsen gleichzeitig**: (a) Rate-Limit (eine tier-basiert 200/50/30, andere hart `>= 20`) · (c) `fee_config`-Lookup (`ORDER BY created_at DESC` vs `club_id NULLS LAST`) · (d) Mengen-Overflow (eine kappt still `p_quantity := v_remaining`, andere lehnt ab) · (e) `price_change_24h` (eine setzt's beim `UPDATE players`, andere nicht). „von-allem-zwei"-Root-Cause. **Regel:** fasst du EINE Achse eines Zwillings-RPC-Paars an, hol BEIDE Live-`pg_get_functiondef` (D87) und diff sie **vollständig** (rate-limit, fee-lookup, qty-handling, alle `UPDATE`-Spalten, guards, side-effects) — nicht nur die Achse im Ticket; sonst fixst du 1 von 4. Konvergenz auf den **kanonischen** Pfad (Mehrheit/`buy_from_ipo`). Mengen-Overflow-Politik (kappen vs ablehnen) = **CEO-Produkt-Entscheidung** (Anil 2026-06-27 = ABLEHNEN, wortgleicher Error-String → identisches `mapErrorToKey`). Beweis = force-rollback Zero-Sum=0 **beide** RPCs + PATCH-AUDIT byte-treu + ILIKE-Konstanten-Audit (Fee 600/150/100, tier 200/50/30). **Falle:** Erklär-Kommentare im neuen Body (`-- vorher: p_quantity := v_remaining`) triggern den eigenen `prosrc ILIKE`-Audit als FALSE-POSITIVE (S406-Falle) → Verhaltens-Smoke autoritativ, nicht ILIKE.

- **Escrow Lock/Unlock-Asymmetrie: Unlock MUSS das Lock exakt spiegeln, Fulfillment KONSUMIERT** (S409): Wenn `create` ein Escrow per `balance -= total, locked_balance += total` setzt (Modell: balance=frei, locked=reserviert, Gesamt=balance+locked), dann MUSS **jeder Refund-Pfad** (cancel, reject, expire, auto-cancel) das exakt spiegeln: `balance += total, locked -= total` (mit `GREATEST(0,…)`). Und der **Fulfillment-Pfad** (Trade kommt zustande) darf NICHT erneut `balance -= total` machen — das Geld verließ balance schon beim Lock und liegt in locked → Fulfillment = nur `locked -= total` (Escrow konsumieren). Zwei Bug-Spielarten, beide Zero-Sum-Brecher: (a) Refund vergisst `balance +=` → Käufer-Geld verschwindet bei Ablauf/Storno (locked reduziert, balance nie zurück); (b) Fulfillment macht `balance -= total` UND `locked -= total` → Doppelbelastung (zahlt 2×). Dritter Klassiker: Refund nutzt `price` statt `price*quantity` → Teil-Release bei qty>1. **Referenz-korrekt = `cancel_offer_rpc`/`reject_offer`; falsch waren `accept_offer` (3 Branches) + `expire_pending_offers`** (Slice 409). Detektion: (1) Live-Probe `create→cancel` muss balance/locked exakt auf Start zurückbringen; (2) force-rollback Zero-Sum `Σ(balance+locked)+Töpfe` vor==nach über create+jeden-Terminal-Pfad (diff≠0 = Bug); (3) Invariante `locked_balance == Σ(price*qty pending escrow-offers)` pro User — ABER die fängt nur locked-Leaks, NICHT balance-Leaks (qty=1-Refund reduziert locked korrekt, balance bleibt leck → separater `offer_unlock`-Tx-Existenz-Check pro Terminal-Offer). Bei jedem neuen Escrow-Feature alle Terminal-Pfade (success/cancel/reject/expire/timeout/insufficient) gegen das Lock spiegeln.
- **Write-only Orphan-Counter neben kanonischem Ledger = „von allem zwei"-Drift** (S406): Eine denormalisierte Saldo-Spalte (`clubs.treasury_balance_cents`), die von mehreren Money-RPCs per `UPDATE … += fee` geschrieben, aber von KEINER Funktion/UI gelesen wird (Auszahlung/Anzeige laufen über einen append-only Ledger + SUM-on-read, D97/S329), driftet still vom Ledger weg (live: 21/30 Clubs, bis 5.715 Credits) — kein aktives Leck (niemand liest sie), aber Integritäts-/Klarheits-Schuld. Detektion VOR „Doppelzählung"-Annahme: `SELECT proname, prosrc ILIKE '%col%' AS writes FROM pg_proc WHERE prokind='f' AND prosrc ILIKE '%col%'` → wenn ALLE Treffer nur schreiben (0 Reader) UND ein paralleler Ledger existiert → Orphan, nicht Doppelzählung. Fix = Writes raus + `DROP COLUMN IF EXISTS` (RPC-Bodies zuerst per CREATE OR REPLACE, dann DROP; geldneutral wenn jede RPC ohnehin eine `trades`-/Ledger-Quelle füttert, z.B. via AFTER-INSERT-Trigger `trg_trades_book_club_treasury`). Beweis = force-rollback Zero-Sum (TOTAL = Σwallets + platform_net + club_ledger_net + Σpbt konstant vor==nach). Audit-Falle: erklärende Removal-Kommentare, die den Spaltennamen wörtlich nennen, triggern den eigenen `prosrc ILIKE`-Audit als False-Positive → Audit auf `non_comment_refs` (NOT ILIKE '%-- [Slice …]%') stellen. Live: Slice 406 (4 RPCs entschlackt, Spalte gedroppt, 3× diff=0).
- **Virtueller Pot aus gespeicherten fee_split-Anteilen → Charge-at-settle braucht ZWEI explizite Topf-Sinks** (S396): Wird ein Event-Pot NICHT vorab escrowed sondern erst beim Settle aus den gesperrten Eintritten gebildet (`type='user'`: Pot = Σ `event_entries.fee_split->>'prize_pool'`, gespeichert beim Lock — NICHT neu rechnen, S356-Konstanten-Drift), MUSS der Settle (a) jeden Eintritt voll abbuchen (`balance/locked −= amount_locked`), (b) den **FLOOR-Rundungs-Rest** (`pot − distributed`, entsteht durch ROUND in rank_rewards + FLOOR per_person) → Topf buchen, UND (c) den **0-Gewinner-Pfad** (`v_scored_count=0`: Eintritte da, aber keine Lineups) → ganzer Pot → Topf. Fehlt (b) oder (c) → Geld charged-aber-nicht-ausgezahlt = vernichtet (Zero-Sum-Bruch). Zero-Sum-Beweis: `SUM(wallet.balance) + topf_saldo` vor==nach über create+settle (force-rollback `BEGIN; DO; RAISE`). Additiver Branch in der bestehenden Settle-RPC strikt `IF type='user'`-guarden → non-user byte-identisch (PATCH-AUDIT). Owner-Trennung: settle=„ended", eigener Cancel-RPC=„cancelled", Escrow-Trigger no-op (kein Branch trifft + `prize_escrowed=false`). Live: Slice 396 (force-rollback diff=0 in 3 Configs).
- **Latente CHECK-Lücke wird erst vom ersten echten Money-Pfad gefeuert — force-rollback-Smoke deckt sie auf, nicht tsc/CI** (S396): Ein `transactions.type` (oder source/status), den eine RPC seit jeher schreibt, der aber NIE im CHECK stand, bleibt unsichtbar solange der Pfad in Prod nie lief (`event_entry_lock`: scout_events_enabled=false; `fantasy_reward`: alle Events prize_pool=0). Erst der erste echte Aufruf wirft `23514`. Beim Bauen eines Money-Features, das einen bisher dormanten Pfad aktiviert: VOR dem Smoke `pg_get_functiondef` aller berührten RPCs nach `INSERT INTO transactions`/`book_*`-`type`/`source`-Literalen greppen und gegen den Live-CHECK (`pg_get_constraintdef`) diffen — nicht nur die neu hinzugefügten Typen. Der Smoke fängt es deterministisch; tsc/Unit NIE (CHECK ist DB-seitig). Mit-gefangen: `chk_event_type` auf `event_fee_config` (zweiter CHECK auf derselben Spalte, von der Impact-Analyse übersehen). Verallgemeinert S330/S359/S379: eine Enum kann durch mehrere unabhängige Gate-Flächen gegatet sein — alle enumerieren.
- **Scope-Achse (z.B. league_id) additiv auf Aggregat-Snapshot/Winner-Tabelle → globalen Insert auf `WHERE <achse> IS NULL` + UNIQUE `NULLS NOT DISTINCT`** (S383): Bekommt eine Money-Snapshot/Winner-Tabelle mit bestehenden GLOBALEN Aggregat-Inserts eine optionale Scope-Spalte (global=NULL, scoped=UUID), MUSS (a) der bestehende globale `INSERT … SELECT` um `AND <scope> IS NULL` ergänzt werden — sonst zieht er die später eingefügten scoped-Zeilen mit (Doppelzählung/falsche Winner); (b) die UNIQUE-Constraint die Achse aufnehmen mit `NULLS NOT DISTINCT` (PG15+) — sonst kollidieren scoped-Zeilen gleicher (month,dim,rank) ODER die globale Idempotenz (zwei NULL-Zeilen) bricht. Zwei-Phasen-Befüllung (global-Loop, dann scope-Loop): scope-Loop VOR den Coverage/Debit-Check legen, damit `SUM(reward) WHERE month=…` global+scoped in EINER Summe deckt + EIN Debit zero-sum bleibt. Live: `monthly_liga_snapshots/_winners` + `close_monthly_liga` Pro-Liga-Manager-Payout (E-2b).
- **Config-Wert steuert Geld-/Tally-Pfad → Recalc-on-Save im Write-RPC** (S347): Ein Config-Wert, der einen **gespeicherten** money/tally-relevanten Ableitungswert beeinflusst (z.B. Tier-Schwellen → gespeicherter `rank_tier` → Poll-Stimmgewicht), → im Write-RPC nach dem Config-UPSERT **synchron** alle betroffenen abgeleiteten Zeilen neu rechnen (fail-isolierte `FOR … LOOP BEGIN … EXCEPTION WHEN OTHERS THEN CONTINUE`), Erfolgs-Count zurück. Nicht „nächster Cron".
- **UI-Gate vs. RPC-Gate-Drift bei Platform-Admin-Override** (S347): Spiegelt ein UI-Gate einen synthetischen Platform-Superadmin-Override (`is_admin=true`), MUSS der Write-RPC denselben Bypass tragen: `IF NOT EXISTS(profiles top_role='Admin') AND NOT EXISTS(club_admins role IN('owner','admin')) THEN RETURN {success:false,'not_club_admin'}`. Sonst Button-zeigt-aber-RPC-rejected.
- **Bank-Ledger `balance_after`: NIE „letzte Zeile" via created_at/id — SUM unter Row-Lock** (S329): Vorigen Saldo NIE als `ORDER BY created_at DESC, id DESC LIMIT 1` lesen (same-txn = gleiche created_at, uuid-id-Tiebreaker ist kein Insert-Order). Stattdessen `PERFORM 1 FROM accounts WHERE id=p_id FOR UPDATE` + `v_prev = SUM(CASE direction credit/debit)` — race-frei + robust gegen Bulk-Buchungen.
- **SUM(bigint) = numeric → Cast-Trap bei Funktions-Signatur** (S329): `SUM(bigint_col)` liefert `numeric` → `PERFORM fn(…, SUM(x), …)` failt (`function fn(numeric) does not exist`). Fix: `COALESCE(SUM(x),0)::bigint`; untypisiertes NULL → `NULL::uuid`. apply_migration fängt es als harten Rollback (Verify-Gate).
- **UNIQUE-/Kardinalitäts-Flip einer Tabelle → ALLE SQL-Reader auf Row-Fanout auditieren, nicht nur Services** (S419): Ändert eine Migration den UNIQUE-Schlüssel einer Tabelle so, dass mehr Zeilen pro altem Schlüssel entstehen können (`player_gameweek_scores`: `UNIQUE(player_id,gameweek)` → `(player_id,fixture_id)` + `league_id` denorm → ein Spieler kann >1 Zeile pro (player,gameweek) haben), bricht JEDER Reader still, der die alte 1:1-Annahme trägt — besonders `LEFT JOIN <table> ON <alte_unique_cols>` ohne die neue Diskriminator-Spalte = **Row-Fanout** im `jsonb_agg`/Result (Duplikat-/Fremd-Slots). Die Code-Reading-Liste fängt typisch nur TS-Services + den offensichtlichen Money-RPC; **live-only RPCs in `supabase/migrations/` fallen durch** (Reviewer-Catch S419: `rpc_get_recent_player_scores`/Form-Bars, Slice 274). **Pflicht-Audit bei jedem Schlüssel-Flip:** `grep -rl "<table>" supabase/migrations/` + jede schreibende/lesende RPC live (`pg_get_functiondef`, D87) auf `JOIN <table>` ohne neue Spalte prüfen. Fanout-sicherster Fix = Skalar-Subquery mit `SUM(... WHERE <neue Spalte>)` statt JOIN (eine Zeile pro Outer-Row garantiert, aggregiert konsistent zum Money-Reader). **Latenz-Falle:** der Fanout zündet erst, wenn der NEUE Writer wieder läuft (Backfill hielt anfangs 1 Zeile/altem-Schlüssel) — force-rollback-Smoke mit Probe-Insert der Mehrfachzeilen deckt ihn auf, ein Live-Call direkt nach Migration NICHT. DB-Schwester von errors-frontend S414/415/416 („von-allem-N"-Surfaces). Live: Slice 419 (fixture-bound scores) + 419b (Form-Bar-RPC Skalar-SUM-Heal, old_dup_pairs=1→new=0).
- **Multi-Hop Cron-Bridge ohne Trigger: rating→fantasy_points→gw_score Divergenz** (S313): Semantik per RPC-Bridge (nicht Trigger) über mehrere Tabellen propagiert → jede out-of-band-Mutation der Quelle (Backfill, MCP-SQL) desynct Downstream still. Bei manuellem Eingriff in `fixture_player_stats.rating`/`fantasy_points` → `sync_fixture_scores(p_gameweek)` als Pflicht-Folgeschritt.
- **Leere Backfill-Platzhalter sehen aus wie „Balances ohne Audit-Trail"** (S306): „N Balances + 0 Ledger-Rows = Risiko" ist FALSCH, wenn die N Zeilen leere Platzhalter sind (alle balance/earned/spent=0, identisches updated_at = 1 Batch). Vor jeder „Aggregat ohne Audit = P1"-Klassifikation prüfen: (1) Werte echt? (2) Write-RPCs loggen in Ledger? (3) gibt's Aufrufer im App-Code?
- **Seed-Wert-Poisoning in Fallback-Formel-Branch** (S303): Plausibler Seed (`last_price=10000` für alle ungetradeten) vergiftet eine Formel mit Fallback-Branch (`… → last_price>0 → use`) → naiver Recompute-Backfill überschreibt korrekte Werte mit Müll. Regel: **Hygiene VOR Formel-Vertrauen** — Source-lose Zeilen auf „nie-X"-Sentinel (`0`) normalisieren (`NOT IN (SELECT fk FROM source WHERE fk IS NOT NULL)`), DANN recompute.
- **Bulk-Backfill verbrennt Set-once-Trigger-Sentinel** (S368e): Ein Data-Repair-`UPDATE` auf eine denormalisierte Spalte, deren `NULL`-Zustand als „noch nie gesetzt"-Sentinel eines `set-once`-Triggers dient (`UPDATE … WHERE col IS NULL`), setzt den Sentinel für noch-leere Zeilen vorzeitig → der Trigger feuert beim echten Erst-Event nie mehr (Wert bleibt der Backfill-Default). Vor jedem Backfill einer trigger-gepflegten Spalte: prüfen ob ein anderer Trigger ihren NULL-Zustand als Sentinel nutzt → Backfill auf Zeilen MIT bereits eingetretenem Quell-Event beschränken, sentinel-relevante Zeilen NULL lassen (`WHERE EXISTS (SELECT 1 FROM <source> …)`). Live-Beispiel: 368e setzte `initial_listing_price=MV/10` auch für IPO-lose Spieler → `trg_set_initial_listing` (`WHERE ilp IS NULL`) hätte beim ersten echten IPO nicht mehr gegriffen; Heal = Sentinel-Restore `SET ilp=NULL WHERE NOT EXISTS(ipos)`.

## Supabase / Postgres

- **Tenant-Window Achsen-Erweiterung: Per-Player vs. Per-Liga** (S274): „Letzte N pro Tenant"-Aggregat hat mehrere Tenant-Achsen (Player vs. Liga) mit gegensätzlichen Effekten. Decision-Tree: bei Aggregat-Drift in der Achse (Liga-Lag) ERST Drift fixen, dann absolutes Window. Performance-Trap: Differential-`LEFT JOIN fps + GROUP BY` kostet 8× — Pragma `NULLIF(score,0)` ist drop-in same-cost.
- **PostgREST RPC-Pfad ignoriert `.range()` und `?limit`** (S270d v2): Bei `TABLE`-Return-RPCs cappt PostgREST hart bei 1000 Rows — `.rpc('fn').range(0,99999)` / `.limit(99999)` werden ignoriert (`content-range: 0-999/*`). Fix: RPC auf `RETURNS JSONB` + `jsonb_agg(...)` Single-Row umstellen; Service parst `data as Array<…>`. Pre-BUILD: `EXPLAIN ANALYZE` → >1000 rows → JSONB-Return.
- **History-Gap-Tag-Sensitivität bei strict-7d-LEFT-JOIN** (S271): Trend-Calc mit `past.date = CURRENT_DATE - INTERVAL '7 days'` → fehlt EIN Snapshot-Tag im Fenster, wird `past.x IS NULL` → ALLE Trends NULL für den Tag (Cron-Log bleibt `success`). Fix: `LEFT JOIN LATERAL (… h.date BETWEEN CURRENT_DATE-14d AND CURRENT_DATE-3d ORDER BY date DESC LIMIT 1)`.
- **Per-Tenant-Window vs. Global-MAX in Aggregat-Services** (S270): Multi-Tenant-Service mit globalem `MAX(time_col)` als Fenster-Anchor → Tenants mit Lag bekommen leere Slots trotz vorhandener Daten (Süper Lig GW37, EU GW32 → 5/5 NULL). Fix: server-side Per-Tenant-`ROW_NUMBER() OVER (PARTITION BY tenant ORDER BY time DESC)` via RPC (+ AR-44 REVOKE/GRANT). Nicht client-side group-by (60k rows transfer).
- **Migration-Heal v1→v2 Same-Session** (S207): Same-Session falsch-applied Migration → v2 mit GLEICHEM Filename-Stem + Timestamp-Bump (+5 min) drüber. Voraussetzung: v1 nutzt `CREATE OR REPLACE` (idempotent) + gleiche Signatur. Hatte v1 Schema-Changes (CREATE TABLE/INDEX) → v2 braucht DROP IF EXISTS. Verify: `pg_get_functiondef` matcht v2-Logic. NIE gleicher Timestamp (Tracker sieht nur 1 File).
- **Same-Day-Migration mit FRÜHEREM Timestamp als Vorgänger-Slice** (S326): Späterer Slice mit niedrigerem Filename-Timestamp als früherer → Live-DB korrekt, aber `db reset`/greenfield läuft Filename-Order → älterer überschreibt neueren (bei Signatur-Change: koexistierende Overloads). Fix: späteren Slice immer auf höheren Timestamp `git mv`. Detection: `grep -l "CREATE OR REPLACE FUNCTION public.<name>" supabase/migrations/*.sql | sort`.
- **Kanonische abgeleitete Spalte: `GENERATED ALWAYS AS (immutable_fn(raw)) STORED` statt Backfill+Trigger** (S391): Soll eine dreckige Stamm-Spalte (mixed Schreibweisen, z.B. `players.nationality` = Türkiye/Turkey/TR) für Regeln/Joins kanonisch werden UND die Rohspalte erhalten bleiben → eine **generierte Spalte** aus einer **IMMUTABLE** Mapping-Funktion ist der zero-drift/zero-trigger/zero-backfill-Weg (DB rechnet ISO immer aus dem Roh-Wert, Scraper schreibt weiter Rohnamen, kein D39-Trigger, kein GUC-Bypass, kein Sentinel-Burn-Risiko S368e). Erbt automatisch die Tabellen-RLS (keine neue Policy). Voraussetzung: Funktion wirklich IMMUTABLE (reine CASE/VALUES, kein now()/Tabellen-Read) — sonst lehnt `GENERATED` hart ab. Partial-Index `WHERE col <> ''` für die Folge-Regeln. Vorzug vor in-place-Backfill (destruktiv, Scraper-Konflikt). Live: Slice 391 (4349 mapped, unmapped=0, TR-Bucket 4 Schreibweisen→762 vereint). **TS↔SQL-Mapping-Duplikat = Divergenz-Vektor (S391 Reviewer):** Wird eine Display-Util-Map (`src/lib/utils/countryNameToIso.ts`) nach SQL portiert (für DB-Regeln), driften Display (TS) und Regel (SQL) bei künftiger Map-Änderung leicht auseinander → bei jeder Änderung BEIDE pflegen (Kommentar in beiden Files + idealerweise CI-Parität-grep).
- **CREATE OR REPLACE FUNCTION — PATCH-AUDIT PFLICHT** (S156 FAIL): Body-Rewrite einer SEC-DEFINER-RPC → NIE vom ersten Create ableiten. Baseline = **live** `pg_get_functiondef('public.<name>(args)'::regprocedure)` (D87), nicht die alte Migrationsdatei (kann stale sein, z.B. `calculate_fan_rank` lebt nur live). Sonst Silent-Revert aller Zwischen-Patches (auth.uid()-Guard, fee_config, min_tier). Post-Apply: `pg_get_functiondef() ILIKE '%<guard>%'` pro Feature. **Konstanten-Audit (S356):** ILIKE-Präsenz-Checks (`%book_club_treasury%`, `%v_weight%`) fangen KEINEN Wert-Drift (`* 70` vs `* 80`). Money-RPC-Patch-Audit MUSS Fee-/Money-**Konstanten** gegen `trading.md` asserten (`def ILIKE '%* 80) / 100%'`). Live-Beispiel: Slice 343 rekonstruierte `cast_community_poll_vote` aus der `slice_336`-**Datei** statt Live → revertierte 337s Fee 80→70 still, ~5 Tage live unbemerkt (in S356 geheilt).
- **UNIQUE/Kardinalitäts-Flip → ALLE Writer auditieren, nicht Conflict-/File-Grep** (S453): Ändert eine Migration den UNIQUE-Key einer Tabelle (z.B. `player_gameweek_scores` `(player_id,gameweek)`→`(player_id,fixture_id)`, S419/D113), MÜSSEN ALLE schreibenden Funktionen mitgezogen werden — sonst `42P10` (no matching ON CONFLICT) + NOT-NULL beim ersten echten Lauf, Off-Season-maskiert (BEFORE-Live: `admin_resync_gw_scores(26)`→42P10). Completeness = **pg_proc-Writer-Enumeration**, NIE (a) `supabase/migrations/`-File-Grep (live-only Fns unsichtbar) NOCH (b) formatierungs-sensitiver `ON CONFLICT(...)`-ILIKE (verpasst delegierende/abweichende Writer): `SELECT proname, prosrc ~* 'on conflict.*fixture_id' AS ok FROM pg_proc WHERE prokind='f' AND prosrc ~* 'insert into <table>'`. Jede Row muss neu-konform sein ODER an die kanonische Writer-Fn delegieren (S453: `admin_import_gameweek_stats` rief `sync_fixture_scores()` = safe, der Conflict-Grep hätte ihn fälschlich verdächtigt). Fix = stale Writer EXAKT auf die live-korrekte Referenz-Fn spiegeln (nicht neu erfinden, force-rollback-Smoke); 3-Wege-Write-Dup als Residual tracken (Forward-Drift-Vektor: 419 heilte nur 1/3).
- **Werte-Skala-Flip → ALLE Downstream-Trigger/Reader DERSELBEN Spalte mit-auditieren** (S454): Ändert eine Migration nicht den TYP, sondern den **Wertebereich** einer Spalte (z.B. `user_stats.total_score` von gedeckelt-3000 auf uncapped scout-Summe), MÜSSEN alle Downstream-Trigger/Reader, die auf der alten Skala kalibriert sind, mit-auditiert werden. Hier: `trg_sync_level` (AFTER UPD OF total_score) rechnet `calculate_level_from_score` + schreibt **USER-NOTIFICATIONS** → ein naiver Backfill hätte 70× „Aufstieg!" gespammt (irreversibel). Schwester zu S419 (Kardinalitäts-Flip→alle Reader) + S453 (Writer-Enum) auf der **Werte-Achse**. Drei Folge-Regeln: (a) **Backfill-UPDATE auf eine Tabelle mit AFTER-Notification-Trigger → Trigger temporär `DISABLE`/`ENABLE` (transaktional, rollt bei Fehler mit zurück)**, sonst spammt die Migration User-facing Notifications; den legitimen Seiteneffekt (hier `profiles.level`) direkt + still nachziehen. (b) **`ALTER COLUMN ... TYPE` blockt mit `0A000` wenn ein Trigger die Spalte in `UPDATE OF col` referenziert → DROP+recreate den Trigger um den ALTER**; Dependency-Check (Views/Generated/Trigger/Constraints) VORHER, nicht iterativ scheitern. (c) Denorm-Projektion (user_stats ← scout_scores) = legitimer Denorm-mit-Trigger (Register §4) NUR mit Trigger auf der Quelle (jede Quell-Änderung propagiert); ohne = trigger-lose Drift = krank.
- **Trigger+GUC-Invariant-Enforcement — generalisiert** (D39): DB-Level Data-Integrity-Invariant über mehrere Code-Pfade (Scripts/Crons/RPCs/MCP-SQL) → `BEFORE <OP>` Trigger mit (1) Escape-Hatch `current_setting('bescout.allow_<f>',true)='true'` (2) NULL-Guards (3) Invariant-Check `RAISE EXCEPTION`.
